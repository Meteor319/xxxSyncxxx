📄 AI 資源管理特工 (Resource Agent) 系統規格說明書1. 背景與痛點 (Context & Pain Points)角色場景： 在高精密、高人力的 IC 設計環境中，技術主管需精準掌握研發資源。原始資料： 目前仰賴散落的 Excel（每周 Timecard 與每季 Workload），缺乏即時連動。核心痛點： 1.  資料滯後： 傳統手動彙整無法在資料更新後立即發現異常。2.  分配不均： 難以跨專案動態識別誰過載、誰閒置。3.  分析成本： 主管需花大量時間比對數據，而非思考調度決策。2. 解題邏輯：AI 時代的通用化工具 (Core Philosophy)本系統不採用「純 AI 讀檔」模式（避免 Copilot 算術錯誤），而是採取：「確定性引擎 (Deterministic Engine) + 語義解釋器 (Semantic Interpreter)」確定性 (Python/SQL)： 所有的時數加總、百分比偏離、人力缺口，由 Python 100% 精確計算。語義化 (AI Agent)： 計算結果轉為 JSON 後，由 AI 根據 CLAUDE.md 的規則產出具備「主管決策高度」的建議文字。通用化 (Agent Skills)： 功能模組化為獨立 Skills，透過 Gitea 管理規則，確保系統易於維運且具備跨資料源擴充性。3. 系統架構圖4. 技術規格 (Technical Specifications)4.1 數據中台 (Data Foundation) - MariaDB採用星狀模型確保資料正確性：維度表 (dim_)： 包含 Employee (含 Skillset 標籤) 與 Project (優先級)。事實表 (fact_)： 紀錄 Timecard (實際) 與 Workload (計畫)。稽核表 (sys_log)： 紀錄每次 Excel 匯入的時間與筆數，確保數據可追溯。4.2 診斷規則庫 - CLAUDE.md這是系統的「大腦規範」，存於 Gitea，定義：異常判定指標：專案偏離度 ($D_p$)： $\frac{\text{Actual} - \text{Plan}}{\text{Plan}} > 25\%$。個人負載率 ($U_i$)： $\frac{\text{Actual Weekly Hours}}{40} > 120\%$。調度邏輯： 偵測到缺人時，自動檢索 skillset 匹配且 $U_i < 80\%$ 的可用資源。5. 核心功能模組 (Agent Skills)技能名稱技術實現數位化任務Data Ingestpandas + sqlalchemy自動讀取 Excel，完成欄位 Mapping 並寫入 MariaDB。Logic AnalyzeSQL View + Python執行精確計算，產出結構化 Anomaly JSON 報告。Decision PromptCopilot / LLM讀取 JSON 並結合 CLAUDE.md 邏輯，生成自然語言調度建議。Auto DispatchOffice365 API將結果同步更新至 SharePoint List (存檔) 與 Wiki (展示)。6. 管理介面：Streamlit Agent Cockpit為主管量身打造的控制中心：上傳端： 主管拖放 Excel，系統自動執行清洗與計算。視覺化： 使用 ECharts 動態展示「專案偏離紅綠燈」與「人力池負載」。決策確認： AI 產出建議（例如：將 A 調往專案 B），主管點擊「確認發布」後即完成全流程。7. 未來擴充性 (Scalability)跨資料源： 預留介面可串接 Gitea Commit 次數，判斷工時增加是否伴隨產出增加（診斷技術瓶頸）。自動化升級： 未來可轉為 Gitea Actions 觸發，達成「檔案上傳即通報」的全自動化維運。💡 點子發想者的最後小語這份規格書將你的**專業技術（Python/DB）與管理思維（Agent/CLAUDE.md）**完美結合。它不再只是一個小工具，而是你團隊中的「虛擬副手」。
