OMO/WPO × AI 知識管理

讓 AI 真正讀懂我們的業務
而不只是回答通用問題

一套以 Token 效率為核心的三層知識架構，將散落在 100+ 份文件中的組織智慧，轉化為 Claude 可反覆精準調用的結構化知識庫。

為何適合 OMO

這個架構是為 OMO 的工作特性量身設計的

OMO/WPO 的知識環境有三個顯著特徵，一般的「全文件丟給 AI」做法無法有效應對。我們的框架針對這三個特徵各有對應設計。

◆
跨領域、高交叉
營運流程（HC、M Time）與技術 Milestone（MRD→CMP→MP）高度連動。單一領域的知識庫無法覆蓋日常工作的真實問題。

對應設計：三個領域各一張知識卡 + 跨域橋接機制，讓 Claude 能同時理解營運時程和技術節點的銜接。
◆
術語密度高
組織內部大量使用縮寫（OMO、WPO、RAS、CMP...），如果 AI 不理解這些術語，每次回答都可能出現低級錯誤。

對應設計：L0 常駐術語表，每次對話自動載入，確保所有回答都使用正確的組織語言。
◆
圖表承載關鍵資訊
Milestone 流程圖、HC 分配圖表、組織架構圖——大量關鍵資訊以圖片形式存在於簡報中，純文字萃取會遺漏。

對應設計：圖片→文字描述、流程圖→Mermaid 語法的標準化轉換流程，一次轉換、永久複用。

推廣策略

從 OMO 驗證到跨單位複製

整套框架設計為「一次驗證、可複製推廣」的模式。OMO 先跑通，其他單位只需要替換領域內容，不需要重新設計架構。

推廣路徑

OMO 試點驗證 → 產出標準範本 → 其他單位複製 → 跨單位知識互通
對其他單位的價值
每個單位都有自己的術語、流程、文件。本框架的三層架構是通用的——只要替換 L0 術語表和 L1/L2 的領域內容，同樣的 Token 節省和品質提升就能在新單位重現。
推廣門檻低
完整的 XML 指令集 + 空白模板 + 操作指南已備妥。新單位不需要技術背景，只需要一位了解業務的同仁負責文件收錄，Claude 會自動執行品質檢查和迭代優化。

關鍵優勢：當多個單位都採用相同架構時，未來有機會實現跨單位的知識互通——A 單位的 Claude 可以參考 B 單位的 L1 知識卡，減少跨部門溝通的資訊落差。

持續優化

內建的自我進化機制

這不是一個做完就放著的系統。框架內建了三個持續優化的閉環，讓知識庫越用越精準。

閉環 1：收錄即驗證
每收錄 5 份文件，自動執行品質檢查並記錄指標。L1 知識卡的覆蓋率、術語表完整度、交叉引用正確性都有量化追蹤。
閉環 2：缺口即收錄
Claude 在回答問題時如果發現知識缺口，會主動標記「缺少 XXX 資訊」。這些標記匯集後，就是下一批收錄的優先清單。
閉環 3：使用即優化
高頻被問到的問題會回饋到 L1 的 Q&A 區塊，下次同樣的問題就能用更少的 Token 更快回答。知識庫會自然地向「最有價值的方向」濃縮。

迭代收斂流程

第 1-2 輪：校準期
收錄前 10 份文件，確認 L0/L1/L2 的分層是否合理、術語表是否完整。預期會有較多調整。
第 3-4 輪：穩定期
收錄流程趨於穩定，L1 覆蓋率達到 4/5 以上。開始有團隊成員能獨立操作。
第 5 輪以後：收斂與維運
品質指標連續達標，框架收斂。轉入日常維運模式——新文件即時收錄，每月審閱一次 L1 知識卡。
實施細節

具體怎麼做？從哪裡開始？

以下是完整的實作步驟。每一步都有明確的輸入、操作和產出，任何了解業務的同仁都能按照這個流程獨立執行。

0 環境準備：建立 Claude 專案 30 分鐘 · 一次性

在公司的 Claude 平台上建立一個專屬 Project，作為知識庫的「家」。所有收錄的知識都會存放在這裡，團隊成員共用同一個 Project。

0-1
建立新的 Claude Project，命名為 「OMO 知識助手」（或你們偏好的名稱）
0-2
將 XML 指令集（已備妥）貼入 Project 的 Custom Instructions。這是 Claude 的「行為說明書」，告訴它如何處理文件、怎麼分層、品質標準是什麼
0-3
將三份空白模板上傳為 Project Knowledge：L0_index.md（知識索引）、L1_tech_card.md（技術知識卡）、L1_ops_card.md（營運知識卡）、L1_mgmt_card.md（管理知識卡）

完成標誌：開一個新對話，對 Claude 說「請展示目前的知識索引」，它會回覆一份空的索引結構——代表環境就緒。

1 第一份文件：組織縮寫與架構表 1 小時 · 建議第一天

刻意選擇「組織縮寫表」作為第一份收錄文件，因為它會被提升到 L0 常駐層，之後所有對話都受益。這是投資報酬率最高的起點。

1-1
上傳組織縮寫/架構文件到對話中
1-2
對 Claude 說：
── 操作指令（直接複製貼上即可）──

請按照 ingestion_protocol 收錄這份文件。
這是組織架構與縮寫對照表，請特別注意：
1. 執行 L0 提升測試，將高頻術語收入常駐術語表
2. 如果有組織架構圖，請轉為文字描述
1-3
Claude 會自動執行：萃取內容 → 判斷哪些術語該進 L0 → 產出 L2 完整萃取 → 更新 L0 索引和術語表。你只需要檢查它的產出是否正確（縮寫有沒有對、架構有沒有漏）
1-4
確認無誤後，將 Claude 產出的更新版 L0_index.md 替換 Project Knowledge 中的舊版

完成標誌：開一個新對話，隨意用幾個部門縮寫問問題，Claude 應該能自動使用正確的全名回應。

2 收錄第二份：Modem 開發流程 (Project Milestone) 2-3 小時 · 建議第一週內

這份文件是框架的真正壓力測試——它包含流程圖（需轉 Mermaid）、多階段定義（需結構化）、可能還有圖表（需文字描述）。成功處理這份，代表框架能應對最複雜的文件類型。

2-1
上傳 Milestone 文件，對 Claude 說：
請按照 ingestion_protocol 收錄這份文件。
這是 Modem 產品的技術開發流程 (Project Milestone)，請特別注意：
1. 所有流程圖請轉為 Mermaid 語法
2. Milestone 各階段的定義請完整保留
3. 圖表中的數據請重建為表格
2-2
Claude 會產出 L2 完整萃取（含 Mermaid 流程圖）。你需要驗證：流程圖的節點和連線是否正確、每個 Milestone 的定義是否完整、有沒有遺漏重要資訊
2-3
Claude 會同步更新 L1 技術知識卡，加入流程摘要和高頻 Q&A。檢查 L1 的摘要是否抓到重點——如果你覺得「看 L1 就能回答大部分關於 Milestone 的問題」，就是成功的
2-4
確認後，更新 Project Knowledge 中的 L0、L1、並新增 L2 檔案

完成標誌：問 Claude 「MRD 到 MP 的主要 Milestone 有哪些？」——它應該能只用 L1 就正確回答。再問「CMP 階段的具體驗證項目有哪些？」——它應該會告訴你需要載入 L2。

3 收錄第三份：組織介紹 1-2 小時

完成三種不同類型的文件（查找型、流程型、背景型）收錄後，就涵蓋了 OMO 最常見的文件型態。此時你對框架的「手感」已經足夠做出初步判斷。

3-1
同樣的流程：上傳 → 指示收錄 → 驗證產出 → 更新 Project Knowledge
3-2
這份完成後，執行 第一輪品質檢查：對 Claude 說「請執行一輪完整的 quality_gate 檢查」。它會自動檢查 L0/L1/L2 的完整度、交叉引用正確性，並回報結果
4 驗證：用真實問題測試效果 2-3 小時 · 建議第二週

收錄完前 3 份文件後，用團隊真正會問的問題來測試。這一步的目的不是「證明框架有效」，而是「找出框架哪裡需要調整」。

4-1
列出 10 個團隊常見問題，涵蓋三種類型：通識（概念問答）、精確（需要具體數字）、產出（需要幫忙寫文件）
4-2
先用舊方法測一遍（手動貼文件給 Claude），記錄每題的準確度（1-5 分）和 Token 用量——這是「改善前的成績單」
4-3
再用新框架測一遍（讓 Claude 從 L0/L1/L2 調用知識），記錄同樣的指標
4-4
對照兩組數據，產出 驗證報告。如果三項指標（準確度、操作時間、Token 消耗）都有改善，就進入擴大收錄階段；如果某項未達標，針對性調整後再測一輪
5 擴大收錄與團隊培訓 Week 3-4

驗證通過後，分批收錄剩餘文件（每批 5 份），同步讓 1-2 位團隊成員學習操作。

5-1
按優先級排序剩餘文件：高頻使用 → 跨部門引用 → 其他。不需要一次全部收錄，先處理最有價值的
5-2
培訓內容只需要三件事：怎麼上傳文件並指示 Claude 收錄、怎麼檢查產出是否正確、怎麼更新 Project Knowledge 檔案
5-3
每 5 份文件收錄完畢，執行一輪品質檢查，記錄到迭代紀錄中，追蹤框架是否趨於穩定
