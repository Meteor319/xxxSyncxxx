# AI 轉型成果盤點 — 思考紀錄與行動建議

> 本文件整理自一場結構化的盤點討論，目的是為今年度 AI 轉型成果做出對內（高層）與對外（thought leadership）兩種敘事，並規劃明年方向。

---

## 一、起點與情境

### 我所在的位置
- **角色**：AI 轉型專案的核心成員
- **觀眾**：對內報給公司高層、對外要對客戶/投資人建立 thought leadership
- **重心**：今年的 AI 轉型偏向**內部效率與流程自動化**
- **最想釐清**：成果如何呈現、該衡量哪些指標、今年做了什麼值得講、明年方向

### 手上的素材
- 失敗或還在試的實驗
- 團隊能力與人才培養成果
- 已上線在用、還在試驗、做到一半的多種狀態並存

### 識別到的張力
手上的素材是「過程導向、學習導向」的故事，但要對的觀眾——尤其是高層——通常想聽「做了什麼、省了多少」。

> **關鍵洞察**：這道張力是機會，不是問題。2026 年市場上充斥著「我們導入 AI，效率提升 X%」的空話，**真正稀缺的是誠實、有深度的轉型敘事**。這恰好就是 thought leadership 的底層燃料。

---

## 二、三條敘事線的初步構想

針對不同觀眾，建議用三條平行但互相支撐的敘事線：

### 給高層的「務實成果版」
- 用「投入產出比」而非單純 ROI
- 把失敗實驗重新框架為 **risk-managed exploration**——強調有方法地試錯
- 團隊培養給具體證據（員工自己長出來的 use case 比 KPI 更有力）

### 給對外的 thought leadership 版
聚焦一個尖銳觀點。候選方向：
- 為什麼我們沒追求 AI 導入速度，而是追求學習速度
- 我們今年砍掉了 X 個 AI 專案，這是學到最值錢的事
- AI 轉型的瓶頸不是技術，是組織的判斷力

### 明年規劃的「下注邏輯」版
不列一堆 initiative，而是給判斷框架：今年學到 A、B、C，所以明年押注 X、不押注 Y，原因是 Z。

## 三、旗艦素材深挖：OMO 的 Claude Code 協作模式

### 核心事實
- 老闆用 Claude Code CLI 為核心，加上外部工具串接，重塑 OMO 單位日常工作流程
- 個人優化執行效率 + 經驗知識留存到 Wiki + 其他同事可參與或透過 Claude Code 直接對接
- 發展時間：兩三個月
- 採用狀況：6-10 人團隊中，**核心幾人重度使用、其他人選擇性使用**
- 量化現況：**目前沒有量化，只有「變快了」的口碑反應**

### 為什麼這個案例重要——三層結構

**第一層：以 CLI 為核心是反直覺的選擇**
2026 年市面上絕大多數企業 AI 導入是「給每個員工一個 ChatGPT/Copilot 帳號」這種個人生產力工具路線。選 Claude Code CLI 作為核心，是選擇了**可程式化、可組合、可串接**的路線——當作組織作業系統在搭，不是給個別員工的玩具。

**第二層：Wiki 作為知識沉澱層**
這個設計補上了大部分企業 AI 導入最大的洞——個人用得很爽，但組織學不到東西。當每個人的 AI 互動都是私人對話，公司沒有任何累積。把經驗結構化進 Wiki、又讓 Claude Code 反過來能讀 Wiki，是一個**知識飛輪**，不是工具導入。

**第三層：協作模式的重塑**
多數公司是「把 AI 加進現有流程」，這裡是「用 AI 重新長出新的流程」——差別非常大。

### 識別到的 yellow flag
- **採用不平均**：核心幾人重度、其他人選擇性。如果現在拿這個對外講 thought leadership，明眼人會看出是「早期採用者效應」，還沒過組織擴散的真正考驗。
- **沒有量化**：「變快了」在 2026 年這個對 AI 已疲勞的市場，說服力有限。

### 但這些 yellow flag 可以反過來用
**因為親身遇到擴散問題，所以開發了一套框架來解決它**——漏洞變成起點。

## 四、補量化的方法（五個月計畫）

### 設計原則
不追求完美數據，追求**有故事的數據**——夠真實、夠具體、能放進簡報和對外文章。

五個月時間夠長，這代表不只是「補量化」，而是要建立一個**持續追蹤的小系統**——把這個案例升級成可以一直講、越講越強的資產。

### 三層量化架構

#### 第一層：自動可拉的客觀數據
從 Wiki 端拉：
- 條目總數、本月新增數
- 被引用/連結次數（內部互引代表知識流動）
- 被 Claude Code 讀取次數
- 作者多樣性（多少不同的人在貢獻）
- **「冷啟動條目」比例**——多少條目來自新進/選擇性使用者，而不是核心幾人

最後一個指標尤其關鍵，直接回答「這個模式能不能擴散」。

從 Claude Code 端拉（如果有 log）：
- 每週 active 使用者數（區分重度 vs 選擇性）
- 平均 session 長度 / 任務數
- 任務類型分佈

> **第一週優先動作**：確認 Claude Code 和 Wiki 的 logging 機制能否打開。這是後面四個月的複利基礎。

#### 第二層：每月輕量自評（5-10 分鐘，固定五題）

1. 本月你用這套系統處理了哪一兩件「以前可能不會這樣做」的事？
2. 本月有沒有一個瞬間是「沒有它就麻煩了」？簡述。
3. 以本月為單位，估計幫你省下多少小時？（憑感覺）
4. 本月你貢獻了幾條 Wiki / 看了幾條別人的？
5. 如果明天這套系統消失，你最不能忍的是什麼？

五個月 × 6-10 人 = **30-50 個故事素材**，足夠支撐高階簡報加對外文章。

#### 第三層：每月一個深度案例（30 分鐘訪談）

訪談對象順序建議：
- M1：選擇性使用者中最積極的——挖「為什麼進來了」
- M2：核心重度使用者——挖最戲劇性的工作流改造
- M3：還沒怎麼用的人——挖阻力（最有價值，敢不敢做是另一回事）
- M4：老闆本人——挖當初決定與現在回看
- M5：新進同事——挖第一眼看到這個系統的衝擊

訪談六題模板：
1. 描述最近用這套系統做的、印象深刻的一件事
2. 同樣的事情，半年前你會怎麼做？差別在哪？
3. 你覺得這套系統做得最好的一件事？最爛的一件？
4. 如果向公司外的朋友解釋這套東西，你會怎麼講？（**這題最關鍵**）
5. 有沒有過「驚訝」的時刻——它做了你沒預期的事？
6. 如果要擴散到更多人，最大的卡點是什麼？

### 五個月節奏

| 月份 | 第一層（自動數據） | 第二層（自評） | 第三層（深訪） |
|---|---|---|---|
| M1 | 建 logging + 基準數據 | 第一次（baseline） | 積極選擇性使用者 |
| M2 | 月度數據 | 第二次 | 核心重度使用者 |
| M3 | 月度數據 | 第三次 | 低使用者 |
| M4 | 月度數據 | 第四次 | 老闆 |
| M5 | **總結報告 + 對外素材整理** | 第五次 | 新進同事 |

## 五、戰略升級：把 OMO 的協作管理知識做成通用框架，推廣到各 RD 單位

### 我問的問題
> 「如果我把目前這整套跟 Claude 協作的管理知識，設計成通用型框架，推廣到各 RD 單位，這算得分嗎？」

### 直接判斷
**算，而且可能是今年最大的得分**——但前提是看清楚這個動作真正的份量。

### 三種定位差別很大

**定位一：把工具推廣出去**（弱）
> 「我們把 OMO 用得不錯的東西，分享給其他 RD 單位試試看。」

聽起來像佈道、internal evangelism，高層會點頭但不會記住。

**定位二：設計通用框架做組織擴散**（中強）
> 「我們把 OMO 實踐萃取成方法論，讓其他單位可以複製。」

已經有料，但仍是傳統的「複製成功經驗」敘事。

**定位三：解決企業 AI 落地的真正瓶頸——知識萃取與組織擴散的方法**（強，thought leadership 等級）
> 「業界都在問為什麼 AI 試點推不開、為什麼大家用得好但組織學不到東西。我們今年用 OMO 當試驗場，發展出一套可複製的『AI 協作管理框架』，現在推到各 RD 單位驗證它的可遷移性。」

### 為什麼定位三在 2026 年特別稀缺
當下整個產業最大的痛點，已經不是「AI 能不能用」，而是**「公司裡有人用得很好，但推不開、學不到、留不住」**——AI adoption gap、productivity paradox、AI scaling problem，叫法不一但本質相同。

我們正在做的，正好是這個問題的解法雛形。而且擁有極稀有的資產：**不是顧問憑空講框架，是從真實實踐中長出來、現在準備拿到其他 RD 單位驗證**。「實踐 → 萃取 → 擴散驗證」的路徑，比任何外部顧問的 PPT 都有說服力。

### 採用定位三要面對的三個現實

1. **框架要能抵抗「OMO 特殊性」的質疑**——其他 RD 單位的工作性質不同，框架要能解釋「在什麼條件下用什麼變體」，不是一套照抄的 SOP。

2. **「推廣」這個動作本身就是研究素材**——別把它當執行任務，當作 N=多個的自然實驗。每個 RD 單位的導入過程都是一個案例，他們適應/改造/拒絕框架的方式會反過來餵養框架本身。

3. **要決定：thought leadership 還是內部影響力**——前者要框架抽象化、能對外講；後者要框架極度貼合公司情境、能落地。最理想是兩者並進，但需要刻意設計。

### 三個我需要先想清楚的問題
1. 這個框架的「核心主張」是什麼？一句話講完，最反直覺、最有觀點的 claim 是什麼？
2. 第一個推廣的 RD 單位怎麼挑？最容易成功的（safe win）vs 最不像 OMO 的（壓力測試）？
3. 老闆怎麼看這件事？把它當「自然下一步」、「OMO 成功擴散」、還是「他想做的更大的局」？

## 六、核心主張的提煉

### 從實踐中挖出的隱含觀點（我認可的部分）
- **觀察一**：AI 轉型不是發工具，是建介面
- **觀察二**：沒有知識飛輪的 AI 導入，是個人加速、組織原地踏步
- **觀察四**：導 AI 不重塑工作流，等於買跑車走原路
- **觀察五**：個人會用到組織會用，中間有道沒人在處理的縫

> 這四個合起來，本質上是一個關於**組織如何學習**的論點，AI 只是觸發點。

### 主張的調性與觀眾
- **調性**：深刻、認真思考過才懂的那種
- **主要觀眾**：C-level 高層
- **設計標準**：90 秒能講完核心，但聽完會想坐下來聊一小時；要有讓人停下來想三秒的反差

### 候選主張

**候選一：AI 轉型的真正瓶頸不在技術成熟度，在組織的學習速度**
個人會用 AI 的速度已遠超組織學習速度，差距越拉越大，導致一個普遍現象——**公司裡每個人都在用 AI，但公司本身沒有變聰明**。

**候選二：我們不是在導入 AI，我們在重新設計組織的「知識代謝」**
傳統組織的知識代謝極度低效——個人經驗死在個人腦子裡、會議紀錄沒人看、SOP 寫完就過期。AI 的真正用途，不是讓個人更快，而是**讓組織第一次有可能擁有真正運作的代謝系統**。

**候選三：給每個員工一個 AI 帳號，是把生產力私有化；建組織的 AI 介面，才是把學習資產化**
100 個 AI 帳號等於 100 個各自加速的孤島；建一套讓 AI 能讀組織知識、組織能從 AI 互動中學習的介面，才把 AI 變成組織資產，而不是個人福利。

**候選四：AI 採用率是個假指標，真正該追蹤的是「組織的提取速度」**
很多公司報告 AI 採用率 80%，但問他們「過去六個月組織因此學到了什麼」，沒人答得出來。我們選擇放棄追蹤採用率，改追蹤——**從一個人會做、到整個團隊都會做，平均要多久**。

### 個人判斷與推薦
- 候選二（知識代謝）**可延展性最強**
- 候選四（採用率假指標）**記憶點最深**

兩者結合的版本：
> **「我們認為 AI 採用率是個假指標。真正該關心的，是組織的『知識代謝』速度——也就是個人經驗變成集體能力的快慢。我們今年沒有追採用率，我們在重新設計這套代謝系統。」**

這個版本同時具備：攻擊性開場（「假指標」讓人停下）、全新框架（「知識代謝」是 fresh concept）、行動承諾（「我們在重新設計」）、可延展（後面接什麼故事都可以）。

## 七、待決事項與下一步

### 需要跟主管對齊的決定
1. **這個案子的定位採哪一級？**——定位一、二、三的選擇，會決定後面所有資源、時間、敘事走向
2. **核心主張採哪個版本？**——候選一到四，或結合版
3. **第一個推廣的 RD 單位怎麼挑？**——safe win vs 壓力測試
4. **是否啟動五個月的量化追蹤系統？**——尤其是 logging 機制要不要先建

### 我認為的關鍵下一步
1. **第一週**：確認 Claude Code + Wiki 的 logging 能否打開，發出第一次自評問卷建立 baseline
2. **第一個月內**：跟主管完成定位與主張的對齊
3. **持續**：把「推廣到 RD 單位」的過程設計成 field study，而非單純 rollout
